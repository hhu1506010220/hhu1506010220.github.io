SPARQL学习笔记（二）

  • Values（赋值）
  • Aggregates（集合）

  • Negation（否定）

    关于solution的筛选是包含两种negation，一种是基于过滤结果的，就是看当前图模式有没有和被过滤的query solution相匹配的；另一种是基于基于删除方法的，即把和其他pattern相关联的solution给一一排除掉。

    FILTER NOT EXISTS

    FILTER NOT EXISTS关键词会测试数据集中是否有数据是不匹配pattern的，并且返回没有匹配成功的数据，不会产生任何额外的绑定(等价于在query pattern已经给定的那些绑定oattern上 去 检验NOT EXISTS里面的pattern没有存在)。

    FILTER EXISTS

    FILTER EXISTS关键词则会测试数据集中是否有数据匹配pattern并返回成功匹配的数据，不会产生任何额外的绑定。

    MINUS

    MINUS关键词相当于是一个选择过滤器，它会同时计算它的左右两边的两个pattern，然后挑选出那些满足左边pattern但是不满足右边pattern的数据(等价于在已匹配的数据上减少匹配，两个pattern相互独立)。

    但是FILTER NOT EXISTS和MINUS也存在一些区别：
  1. 共享变量规则不同。FILTER NOT EXISTS共享变量但是MINUS不共享，比如说假设query pattern的一个变量是?x，FILTER NOT EXISTS里对应的是?a，MINUS里对应的是?b，那么?x和?a实际上是绑定的同一个变量，因为?a是去匹配?x的pattern的，但是?x和?b就不是相同绑定的，因为?x和?b代表了两个独立的pattern，是相互独立的两部分，所以当对于同一个变量graph pattern绑定的是?x、MINUS绑定的是?b的时候，返回的必然是?x。
  2. 作用域不同。FILTER NOT EXISTS的条件是对于每一个query pattern的solution都会去test，并且可以引用每一个solution里面的变量，而MINUS里的条件里是没办法引用query pattern里面的变量的，因此如果它里面的条件用到了query pattern里面的，那么将是无效的。

  • Assignment（重命名）

    表达式的值可以通过将新变量绑定到表达式的值(即RDF项)来添加到solution mapping中。这个变量可以在查询中使用，也可以在结果中返回。
    重命名的使用形式为( expression AS ?var )，有三种使用方式:
    1.使用BIND关键字
    2.SELECT表达式中
    3.GROUP BY表达式中
    如果表达式有错误，那么?var不会和solution绑定，但是查询还是继续。
    BIND关键词允许将一个值分配给一个基本图模式或者属性路径表达式中的变量。使用BIND将结束前面的基本图模式，并且用BIND绑定的这个变量在BIND这个语句出现之前没有在组图模式中使用过。


    此外，还可以用CONCAT(?G, " ", ?S) AS ?name这样的函数来实现变量中间通过空格连接起来的功能。

  • Aggregates（集合）

    Aggregates将表达式应用到solutions组。默认情况下，solution集合只包括一个组，这个组包含了所有的solution。
    分组是用GROUP BY关键字来实现的。
    Aggregates包括了哪些操作呢？主要有以下几个：COUNT， SUM， MIN， MAX，AVG， GROUP_CONCAT以及SAMPLE。通常的使用场景是使用者想要看到关于solution的一些整体性的数据，比如有几个结果、最大的是哪个诸如此类的问题。

    GROUP BY

    在查询的情况下一般不会用到GROUP BY，通常是用SELECT、HAVING、ORDER BY这些关键词，这些操作会隐式地生成一个group，所有solution都属于这个group。GROUP BY就不一样了，它是在solutions的基础上进行操作的，目的是把solution进一步分组。此外，GROUP BY可以和AS，像这样：GROUP BY (?x + ?y AS ?z)，这其实和{ ... BIND (?x + ?y AS ?z) } GROUP BY ?z是等价的。

    HAVING

    HAVING其实和FILTER的作用差不多，但是二者作用的对象不同。HAVING是作用在分好组的groups上的，一般和GROUP BY是连用的；FILTER则是作用在未分组的solution上的。

  • Subqueries（子查询）

    子查询是将SPARQL查询嵌入到其他查询中的一种方法，通常是为了实现那种不得到这个查询结果就无法实现后面的查询的这种查询，例如限制查询中某些子表达式的结果数量。
    在进行子查询运算时，由于SPARQL查询计算的自下而上性质，首先对子查询进行逻辑计算，并将结果投影到外部查询。注意，只有从子查询中投射出来的或者在作用域内的变量才会对外部查询可见。

  • RDF Dataset（数据集）

    RDF数据模型将图表示为由一些三元组组成，这些三元组又由主谓宾语来构成。许多RDF data stores存储了很多RDF图和每个图的信息，从而允许应用程序可以进行涉及多个图的查询。
    SPARQL查询就是在RDF数据集上跑的，每个数据集实际上都是一个图的集合。一个RDF数据集包括一个默认图和 大于等于0个 命名图（命名图由IRI来定义）。一个查询不一定非得匹配默认图，但是可以只设计匹配命名图。
    Active graph是指被用来匹配基本图模式的图，在我们此处提到这个名词之前，之前的笔记里所有的查询都是基于一个简单图，也就是RDF数据集的默认图，来作为我们的active graph的。GRAPH这个关键字就是用来指定命名图中的某一个图做来作为部分查询的active graph的。默认图和命名图将在后面进行讲解。
    RDF数据集的定义并不限制命名图和默认图之间的关系，不同的图的信息可以重复。通常的使用场景如下：
    1.为了在默认图中包含有关命名图的出处信息
    2.为了在默认图中包含命名图中包含的信息

    在RDF数据集里，一种可行的方法就是让默认图RDF merge一些或者所有命名图的信息。在RDF merge里merge和被merge的空白节点不共享。

    指定数据集

    SPARQL查询可以通过FROM或者FROM NAMED关键字来指定（描述）数据集。如果一个查询提供了这样的数据集描述，那么它将会在这个查询服务所出现的所有地方（前提是没有再声明其他的数据集描述）都可以被使用。。RDF数据集也可能是在SPARQL协议请求中指定，在这种情况下，协议描述将覆盖查询本身中的任何描述。如果数据集的描述不被service所接受，那么这个查询请求则会被拒绝。
    RDFmerge的数据集有两大部分，一部分是由FROM声明的默认图，一部分是由FROM NAMED声明的(IRI,graph)对。如果一个查询里没有任何FROM语句但是有FROM NAMED语句，那么我们默认这个图的默认图是一个空图。

    指定Default Graph默认图

    每个From语句会包含一个IRI，用于指定哪个图是默认图。并且这样并不会把这个图变成命名图。如果一个查询提供多个FROM子句来指定了多个IRI，那么默认图则为这些IRIs的RDFmerge后的结果。

    指定Named Graph命名图

    使用FROM NAMED语句来声明命名图，如果同一个IRI被声明了多次，那么结果集中也只会出现一次这个IRI所指代的命名图。FROM NAMED语法表明对于一个IRI而言，我们标识了相应的图，但是RDF数据集中的IRI和这个图之间的关系是间接的。IRI标识了一个资源，而资源由一个图表示(或者更准确地说：由一个序列化图的文档表示)。

  • GRAPH关键字
